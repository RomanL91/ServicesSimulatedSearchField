## Описание директорий

### `core/` - Ядро, конфигурация и контракты

Самый стабильный слой. Не зависит ни от чего. Всё остальное зависит от него.

**`config.py`**
Настройки приложения через `pydantic-settings`. Читает переменные окружения из `.env`. Содержит все параметры: строки подключения к БД, Redis, брокеру, секреты JWT, таймауты и прочие константы. Является единственным источником правды о конфигурации.

**`security.py`**
Алгоритмы безопасности: создание и верификация JWT-токенов, хэширование паролей, проверка прав доступа. Только логика, никаких HTTP-зависимостей.

**`logging.py`**
Настройка логгера приложения. Форматы, уровни, обработчики.

**`interfaces/`**
Абстрактные классы (ABC) для всех ключевых компонентов. Это контракты - определяют что должно быть реализовано, но не как. Сервисный и командный слои зависят только от этих абстракций, никогда от конкретных реализаций.

- `uow.py` - контракт Unit of Work: методы `commit`, `rollback`, вход/выход из контекстного менеджера
- `broker.py` - контракт брокера сообщений: метод `publish(topic, payload)`
- `task_queue.py` - контракт очереди задач: метод `enqueue(task_name, payload)`
- `repositories/` - контракты репозиториев для каждой сущности: методы `add`, `get_by_id` и специфичные для сущности выборки

---

### `domain/` - Бизнес-сущности и бизнес-стратегии

Чистый Python. Нет зависимостей от FastAPI, SQLAlchemy, Pydantic или чего-либо внешнего. Именно этот слой описывает предметную область.

**`user.py`, `order.py` и другие**
Доменные модели - датаклассы с бизнес-методами. Содержат правила и инварианты предметной области. Например: нельзя применить скидку больше 100%, нельзя отменить уже отменённый заказ, нельзя деактивировать уже неактивного пользователя. Это бизнес-правила, а не технические ограничения - они живут именно здесь.

**`strategies/`**
Бизнес-стратегии - классы реализующие паттерн Стратегия для вариативной бизнес-логики. Каждая стратегия - отдельный сценарий обработки запроса. Стратегия принимает UoW, брокер и очередь задач, и сама оркестрирует выполнение команд.

- `base.py` - абстрактный базовый класс стратегии с методом `execute(data)`
- Каждая конкретная стратегия описывает полный сценарий: какие команды запустить, в каком порядке, что делать с транзакцией

Примеры стратегий для заказа: `RegularOrderStrategy`, `PromoOrderStrategy`, `BulkOrderStrategy`. Каждая запускает разный набор команд.

---

### `commands/` - Write-сторона, атомарные операции

Паттерн базируется на принципе CQS (Command Query Separation), который гласит: метод должен либо выполнять действие (команда), либо возвращать данные (запрос), но не оба сразу.

Каждая команда - одна атомарная операция. Принимает необходимые зависимости (uow, broker, task_queue) и выполняет ровно одно действие.

**`base.py`**
Базовый абстрактный класс команды с методом `execute()`. Все команды наследуются от него.

**Типы команд:**

*DB-команды* - работают с UoW и репозиториями внутри транзакции. Добавляют, обновляют, удаляют сущности через `uow.repo`. Должны выполняться до `uow.commit()`.

*Broker-команды* - публикуют события в брокер сообщений (RabbitMQ, Kafka). Принимают `broker` как зависимость. Выполняются строго после успешного `uow.commit()` - иначе при откате транзакции событие уже улетело.

*Task-команды* - ставят задачи в очередь фоновой обработки (Celery, ARQ, своя реализация). Принимают `task_queue`. Также выполняются после коммита.

Структура по сущностям: `commands/order/`, `commands/user/` и т.д. Каждый юзкейс - отдельный файл.

---

### `queries/` - Read-сторона CQRS

Запросы на чтение данных. Не используют UoW и доменные модели - читают данные напрямую через сессию БД и сразу возвращают Pydantic-схему для API.

**`base.py`**
Базовый абстрактный класс запроса с методом `handle()`.

**Принципиальное отличие от commands:**
Queries намеренно проще - нет транзакционной обёртки, нет доменных объектов, прямой SQL или ORM-запрос и сразу DTO. Это ускоряет чтение и позволяет легко оптимизировать выборки независимо от write-стороны. В перспективе read-сторону можно полностью перевести на отдельное хранилище (Redis, materialized view) не меняя контракт.

---

### `infra/` - Инфраструктурные реализации

Всё что взаимодействует с внешним миром. Реализует контракты из `core/interfaces/`. Единственный слой где живут SQLAlchemy, aio_pika, boto3 и другие внешние библиотеки.

**`database.py`**
Создаёт async engine, фабрику сессий, зависимость `get_db()` для FastAPI. Использует строку подключения из `core/config.py`.

**`redis.py`**
Пул соединений Redis. Клиент для кэширования, сессий, pub/sub.

**`uow.py`**
Конкретная реализация `AbstractUoW` на SQLAlchemy. Управляет сессией и транзакцией. При входе в контекстный менеджер инициализирует все репозитории, передавая им текущую сессию.

**`broker.py`**
Конкретная реализация `AbstractBroker`. Например, RabbitMQ через `aio_pika`. Детали подключения берёт из `core/config.py`.

**`task_queue.py`**
Конкретная реализация `AbstractTaskQueue`. Например, Celery или ARQ. При необходимости легко заменяется без изменения команд.

**`models/`**
ORM-модели SQLAlchemy. Описывают схему таблиц. Существуют только в infra и не должны выходить за её пределы. Репозиторий маппит ORM-модель в доменную сущность и обратно.

**`repositories/`**
Конкретные реализации репозиториев на SQLAlchemy. Содержат SQL/ORM-запросы. Маппят результаты в доменные объекты.

**`strategies/`**
Инфраструктурные стратегии - не бизнес, а способы взаимодействия с внешними системами. Например, стратегии отправки уведомлений (email, telegram) или хранения файлов (S3, локально).

**`templates/`**
Реализации паттерна Шаблонный метод для инфраструктурных задач: импорт данных из разных форматов, парсинг, генерация отчётов.

---

### `api/` - HTTP-слой

Тонкий слой. Только HTTP-контракт, валидация входных данных и возврат ответа. Никакой бизнес-логики.

**`v1/endpoints/`**
Роутеры FastAPI. Принимают запрос, валидируют тело через Pydantic-схему, получают нужный Handler через `Depends()`, передают в него данные, возвращают ответ. Один endpoint - 5-10 строк кода.

**`v1/router.py`**
Агрегирует все endpoint-роутеры версии v1. При добавлении v2 создаётся аналогичная структура, старая не трогается.

**`dependencies/`**
Граф зависимостей FastAPI. Здесь происходит сборка: создаётся UoW из сессии, выбирается нужная стратегия на основе входных данных, создаётся Handler и передаётся в endpoint. Это единственное место где знают о конкретных реализациях из `infra/`.

**`schemas/`**
Pydantic-схемы для входных и выходных данных API. Отдельно от доменных моделей. Маппинг между схемой и доменной моделью происходит в endpoint или в handler.

---

### `main.py`

Точка входа. Создаёт FastAPI-приложение, подключает роутеры, настраивает middleware, lifespan события (подключение к БД и брокеру при старте, отключение при остановке).

---

## Правила зависимостей

```
api           →  commands, queries, schemas, domain
commands      →  core/interfaces, domain
queries       →  infra/database, schemas
domain        →  ничего (чистый Python)
core          →  ничего
infra         →  core, domain
```

Стрелка всегда направлена к более стабильному слою. Никогда в обратную сторону.

## Flow запроса

### Write-сценарий: Создание заказа с промокодом

```
1. POST /api/v1/orders
        │
        ▼
2. OrderCreateSchema                        [api/schemas/order.py]
   Pydantic валидирует тело запроса.
   Если данные невалидны - 422 сразу, до бизнес-логики.
        │
        ▼
3. Depends(get_order_handler)               [api/dependencies/commands.py]
   Dependency Injection собирает граф:
   - создаёт AsyncSession из пула
   - создаёт SqlAlchemyUoW(session)
   - создаёт RabbitMQBroker(connection)
   - создаёт CeleryTaskQueue()
   - передаёт всё в OrderHandler
        │
        ▼
4. OrderHandler.handle(data, promo_code)    [api/handlers/order.py или commands]
   Смотрит на входные данные и выбирает стратегию:
   - promo_code == "PROMO20" → PromoOrderStrategy
   - user.is_vip → VipOrderStrategy
   - иначе → RegularOrderStrategy
        │
        ▼
5. PromoOrderStrategy.execute(data)         [domain/strategies/promo_order.py]
   Оркестрирует команды. Контролирует транзакционную границу.
        │
        ├──▶ 6a. CreateOrderCommand.execute(uow)
        │         Создаёт доменный объект Order.
        │         Применяет бизнес-правила (скидка, инварианты).
        │         Передаёт в uow.orders.add(order).
        │         Репозиторий маппит domain → ORM и добавляет в сессию.
        │
        ├──▶ 6b. uow.commit()
        │         Транзакция фиксируется в БД.
        │         Только после этого - внешние вызовы.
        │
        ├──▶ 6c. PublishOrderCreatedCommand.execute(broker)
        │         Публикует событие "order.created" в RabbitMQ.
        │         Другие сервисы могут подписаться.
        │
        └──▶ 6d. ScheduleDeliveryCommand.execute(task_queue)
                  Ставит задачу "calculate_delivery" в Celery.
                  Задача выполнится асинхронно.
        │
        ▼
7. Endpoint возвращает {"id": order_id}     [api/v1/endpoints/orders.py]
```

---

### Read-сценарий: Получение заказа

```
1. GET /api/v1/orders/{order_id}
        │
        ▼
2. Depends(get_order_query_handler)         [api/dependencies/queries.py]
   Создаёт AsyncSession.
   Создаёт GetOrderHandler(session).
        │
        ▼
3. GetOrderHandler.handle(GetOrderQuery)    [queries/order/get_order.py]
   Прямой запрос к БД через сессию.
   Без UoW, без доменных объектов.
   Сразу возвращает OrderReadSchema.
        │
        ▼
4. Endpoint возвращает OrderReadSchema      [api/v1/endpoints/orders.py]
```

Read-путь намеренно короче. Нет лишних слоёв абстракции там где они не нужны.

## Порядок выполнения команд в стратегии

```
async with uow:                        ← открываем транзакцию
    DB-команды (CreateOrder, etc.)     ← пишем в БД
    uow.commit()                       ← фиксируем

Broker-команды (PublishEvent)          ← публикуем событие
Task-команды (ScheduleTask)            ← ставим фоновые задачи
```

Это критично. Если сначала опубликовать событие, а потом упадёт commit - событие уже ушло, но данных в БД нет. Всегда: сначала commit, потом внешние вызовы.

## Принцип расширения

| Что нужно добавить | Что делать |
|---|---|
| Новый юзкейс | Новая стратегия + новые команды |
| Новый endpoint | Новый файл в `api/v1/endpoints/` |
| Новый брокер (Kafka вместо RabbitMQ) | Новый `infra/broker.py`, интерфейс не меняется |
| Новая очередь задач | Новый `infra/task_queue.py` |
| Версия API v2 | Новая папка `api/v2/`, бизнес-логика не трогается |
| Новая сущность | domain-модель + интерфейс репозитория + ORM-модель + реализация репозитория |
| Read-оптимизация | Подменяем только query handler, write-сторона не трогается |

Каждое изменение локализовано. Добавление нового не ломает старое.
